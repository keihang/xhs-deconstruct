# 小红书博主拆解

分析小红书博主的爆款帖，提炼可复用的内容模式。

## 前置条件

### OpenCLI 安装

> [OpenCLI](https://github.com/jackwener/OpenCLI) — 把网站变成 CLI，让 AI agent 操作你的浏览器。

**方式 A：OpenCLIApp（推荐 macOS / Windows）**

下载安装包 → https://opencli.info/download → 安装后打开 app → System 页面安装 `opencli` 命令。

**方式 B：npm 全局安装（Node.js >= 20）**

```bash
node --version
npm install -g @jackwener/opencli
```

### Browser Bridge 扩展

Chrome Web Store 安装 [OpenCLI 扩展](https://chromewebstore.google.com/detail/opencli/ildkmabpimmkaediidaifkhjpohdnifk)，或从 [Releases](https://github.com/jackwener/opencli/releases) 手动安装。

### 验证 + 安装小红书

```bash
opencli doctor                           # 验证连接
opencli install xiaohongshu              # 安装小红书适配器
```

### 其他

- **Obsidian**（可选）：原文文件使用 Obsidian 图片语法，其他 markdown 编辑器也能用
- **网络**：需能访问小红书，建议国内 IP

## 目标

拆解小红书博主，分析每位博主指定时间段内**流量最好的10篇图文帖子**（点赞>100），提炼标题公式、内容结构、封面设计等可复用模式。视频帖子不计入，自动跳过。

## 工作流程

**博主锁定方式（二选一，都用 `/xhs-deconstruct` 触发）**：
- **方式A**：`/xhs-deconstruct 拆解 博主名字` → AI 搜索博主名字
- **方式B**：`/xhs-deconstruct 拆解 https://www.xiaohongshu.com/user/profile/xxx` → AI 直接用链接获取数据


### 第1步：锁定博主

**方式A：`/xhs-deconstruct 拆解 博主名字`**
- 搜索博主名字：`opencli xiaohongshu search "博主名字" --limit 20`

**方式B：`/xhs-deconstruct 拆解 链接`**
- 直接用链接获取数据：`opencli xiaohongshu user "用户发的链接"`

```bash
# 搜索博主名字
opencli xiaohongshu search "博主名字" --limit 20
```

**关键注意事项**：
- 每次搜索间隔**15秒**，避免触发风控
- 搜索结果包含：标题、URL、作者、点赞数、类型（图文/视频）
- 从搜索结果里确认博主是否存在

**如果搜索失败 →**
| 触发条件 | 一线修复 | 仍失败兜底 |
|----------|---------|-----------|
| 返回空结果 | 换关键词重试（如去掉中间字） | 等5-10分钟后再试，告知用户可能触发风控 |
| 返回"安全验证" | 等3分钟重试 | 停止操作，告知用户需要手动完成验证 |
| 网络超时 | 等30秒重试1次 | 告知用户检查网络后重试 |

### 第2步：确认时间段

🔴 **CHECKPOINT：必须先问用户再继续。**

立即询问用户要抓取的时间段：
- 一个月
- 半年
- 一年

**重要**：确认时间段后，再进行后续筛选。不要跳过此步骤直接抓取。

### 第3步：筛选帖子（图文优先）

**目标**：找到该时间段内，点赞>100的**图文帖子**，取前10篇。

**筛选流程**：

1. **先获取帖子列表**（含发布日期）
   - 搜索结果有日期 → 直接用
   - Profile 结果无日期 → 需逐篇读取获取日期

2. **按时间筛选**：只保留用户选择时间段内的帖子

3. **按类型筛选**：只保留图文帖子，跳过视频
   - 搜索结果有类型信息 → 直接用
   - 无类型信息 → 下载第一张图片判断（mp4=视频跳过，jpg=图文继续）

4. **按点赞筛选**：只保留点赞>100的帖子

5. **排序取前10**：按点赞数从高到低，取前10篇

**如果图文帖子不足10篇 →**
| 触发条件 | 处理方式 |
|----------|---------|
| 图文>100赞 < 10篇但 ≥ 3篇 | 有多少抓多少，如实告知用户 |
| 图文>100赞 < 3篇 | 降低门槛到点赞>50，仍不足则告知用户数据量不足 |
| 该时间段内全是视频 | 扩大时间范围或告知用户该博主以视频为主 |

**快速判断类型命令**：
```bash
# 只下载第一张图片（快速判断类型）
opencli xiaohongshu download "帖子URL" --output /tmp/test-download/ --limit 1
# 检查下载的文件格式
ls /tmp/test-download/
# 如果是mp4 → 视频，跳过；如果是jpg → 图文，继续
```

🔴 **CHECKPOINT：筛选完成后，向用户确认筛选结果（帖子列表+数量）再继续下载。**

### 第4步：读取帖子内容并下载原图

用搜索结果里的URL读取每篇帖子的完整内容，并下载所有图片：

```bash
# 读取帖子内容
opencli xiaohongshu note "搜索结果URL" --format json > raw-data/posts/博主名/01-标题.json

# 下载所有图片（不只是封面）
opencli xiaohongshu download "搜索结果URL" --output bloggers/01-博主名/原文/image/01-标题/
```

**关键发现**：
- ✅ 搜索结果URL（带xsec_token）能用
- ❌ explore URL（不带xsec_token）不能用
- 如果URL格式不对，会报错"安全验证"

**读取间隔**：每次间隔**15秒**

**如果读取帖子报错 →**
| 触发条件 | 一线修复 | 仍失败兜底 |
|----------|---------|-----------|
| "安全验证"错误 | 检查URL是否带xsec_token | 跳过该帖，记录到日志，继续下一篇 |
| 返回空内容 | 等3分钟重试 | 跳过该帖，继续下一篇 |
| 下载图片是mp4 | 标记为视频帖子，跳过 | — |
| 连续3篇失败 | 停止，等10分钟 | 告知用户可能触发风控，建议暂停 |

**下载要求**：
- 下载帖子中的**所有图片**，不只是封面
- 图片保存到 `bloggers/01-博主名/原文/image/01-标题/` 目录
- 保持原始文件名（帖子ID_序号.jpg）

**视频/图文判断**：
- 下载时观察文件格式：
  - 如果是 `.mp4` → 视频帖子，跳过，不写原文文件
  - 如果是 `.jpg` 或 `.png` → 图文帖子，继续处理
- 在下载时告诉用户："正在下载，如果是视频会自动跳过"

## 使用示例

用户说：`/xhs-deconstruct 拆解 数字生命卡兹克`

流程：
1. 搜索"数字生命卡兹克"
2. 🔴 立即询问用户："要抓取多长时间的内容？一个月/半年/一年？"
3. 用户选择"半年"
4. 筛选该时间段内的帖子 → 只保留图文 → 只保留点赞>100
5. 按点赞数排序，取前10篇图文（不足就取有多少算多少）
6. 向用户确认筛选结果
7. 逐篇读取内容、下载所有图片、写原文文件（引用所有图片）、写拆解分析
8. 写总结，提炼模式
9. 沉淀发现到skill

│   │   │   │   └── ...
│   │   │   └── ...
│   │   └── 拆解/                 # 每篇笔记的7维度拆解分析

│   ├── posts/                   # 搜索结果 JSON

**关键说明**：
- 原文文件含 frontmatter 元数据（blogger, platform, type, date）
- 图片平铺在 `image/` 目录，文件名格式 `序号-标题.jpg`
- 多图帖子额外文件 `序号-标题_2.jpg`、`序号-标题_3.jpg`...
- 视频帖子的 `.mp4` 文件会出现在 `image/` 中，跳过不写原文
- `raw-data/` 是中间产物，包含 API 原始返回数据，确认无误后可删除
- 拆解文件格式与库森说AI完全一致

## 不要做什么（反例清单）

| # | 不要做的 | 为什么 | 应该做的 |
|---|---------|--------|---------|
| 1 | 不要跳过"确认时间段"直接抓取 | 用户可能只想看近期内容，全量抓取浪费时间且增加风控风险 | 🔴 必须先问用户时间段 |
| 2 | 不要在搜索后立即批量下载 | 短时间大量请求会触发风控，导致IP被封 | 每次操作间隔15秒 |
| 3 | 不要只下载封面图 | 拆解需要分析完整图片结构，只看封面会遗漏关键信息 | 下载帖子中的所有图片 |
| 4 | 不要凭感觉下结论 | 没有数据支撑的分析不可复用，误导后续创作 | 所有结论必须附带数据来源 |
| 5 | 不要在数据不足时硬凑10篇 | 硬凑低质量帖子会稀释拆解价值 | 如实告知用户数据不足 |
| 6 | 不要把视频帖子算进10篇 | 视频和图文拆解维度不同，混在一起会乱 | 只统计图文帖子，视频直接跳过 |
| 7 | 不要同时处理多个博主 | 并行处理增加风控概率，且容易混乱 | 一次只处理一个博主 |
| 8 | 不要用 explore URL（不带xsec_token） | 这类 URL 会触发安全验证，读取失败 | 只用搜索结果返回的 URL |
| 9 | 不要更新 skill 时没有数据支撑 | 没验证的模式会污染 skill 质量 | 更新必须附带数据来源 |
| 10 | 不要省略原文文件的元数据 | 缺少元数据会导致后续检索和分析困难 | 每个原文文件必须有 frontmatter |

## 注意事项

### 文件命名
- 不要用空格，用连字符（-）
- 序号用两位数（01、02、...）
- 标题简短但有意义
- 封面文件和原文文件同名，只是扩展名不同
- **避免命名冲突**：先检查已有博主目录，确保序号不重复

### 风控补充
- 搜索/读取间隔15秒（各步骤已标注，此处不再重复）
- 如果返回空数据或"安全验证"，等几分钟再继续
- 完整 fallback 速查见各步骤的 if-then 表格

## 常见问题

### Q: 搜索结果为空？
A: 可能触发了风控，等5-10分钟再试。或者换个关键词搜索。

### Q: 读取帖子内容报错"安全验证"？
A: URL格式不对，必须用搜索结果URL（带xsec_token），不能用explore URL。

### Q: 下载的图片是mp4而不是jpg？
A: 说明是视频帖子，应该跳过，不写原文文件。这是正常的筛选过程。

### Q: 搜索结果里怎么没有类型信息（图文/视频）？
A: 搜索结果里确实没有类型信息，只有在下载时才能判断。所以需要"先下载试试，视频帖子会自动跳过"。

### Q: 原文文件格式与库森不一致？
A: 检查是否有元数据（blogger, platform, type, date），是否有完整正文，是否引用了所有图片（不只是封面）。

### Q: 拆解分析格式与库森不一致？
A: 检查是否有frontmatter，是否有基本信息表格，是否有6个拆解维度，是否有"为什么有效"分析。

### Q: 图片没有全部引用到原文？
A: 原文文件应该引用所有图片，按顺序排列：`![图1](路径)`, `![图2](路径)`, ...

### Q: 总结提炼不出模式？
A: 先看所有拆解文件，找共性。如果数据不足，如实说明"数据有限，无法提炼可靠模式"。

### Q: 更新skill时不确定是否准确？
A: 所有更新都要附带数据来源。如果没有足够数据，不要更新skill，等下次拆解补充数据。
